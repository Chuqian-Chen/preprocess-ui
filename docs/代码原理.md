# 数据质控 · 预处理工作台 — 代码原理

面向开发/维护者，讲清架构、数据流、关键算法与设计取舍。配套《[使用手册](使用手册.md)》讲操作。

---

## 1. 总体架构

```
┌────────────────────────────────────────────────┐
│  浏览器 (web/index.html + app.js + Chart/Mermaid) │   纯静态，无构建步骤
└───────────────┬────────────────────────────────┘
                │  fetch /api/*  (X-Project-Id 头标识当前项目)
┌───────────────▼────────────────────────────────┐
│  FastAPI (server/app.py)  — 全部路由入口           │
│  ┌──────────────┬──────────────┬──────────────┐ │
│  │ project_mgr  │  qc_engine   │ join_analyzer│ │
│  │ (项目/缓存)   │ (画像/值域)   │ (统计探查)    │ │
│  ├──────────────┼──────────────┼──────────────┤ │
│  │ field_       │ transform_   │  join_ai /   │ │
│  │ preprocess   │ engine       │ ai_transformer│ │
│  │ (清洗流程)    │ (安全执行)    │  (LLM 调用)   │ │
│  ├──────────────┴──────────────┴──────────────┤ │
│  │ ai_settings (配置同源解析)  analysis (分布)   │ │
│  └─────────────────────────────────────────────┘ │
└───────────────┬────────────────────────────────┘
                │  读写文件
┌───────────────▼────────────────────────────────┐
│  workspaces/{项目id}/  raw(只读) / config / output │
│                        profile.json / meta.json   │
└────────────────────────────────────────────────┘
```

**设计原则**：
- **无数据库**：一切以文件系统为真相源。项目 = 一个目录，配置 = JSON，画像 = `profile.json` 缓存。便于备份、迁移、离线。
- **无构建前端**：`web/` 是原生 HTML/JS/CSS，第三方库（Mermaid、Chart.js）本地化到 `web/vendor/`，FastAPI 以 `StaticFiles` 挂在 `/assets`。
- **raw 不可变**：原始数据只读，所有变换写 `output/`，可反复迭代与回退。
- **AI 不执行代码**：LLM 只产出受限的 JSON 操作，由白名单引擎执行（见 §5）。

## 2. 项目与状态模型（`project_manager.py`）

每个项目一个 `workspaces/{12位hex id}/` 目录：

| 路径 | 内容 | 读写者 |
|---|---|---|
| `raw/*.csv` | 导入的原始表（只读） | 创建时写入，之后只读 |
| `config/schema_overrides.json` | 字段 → 类型覆盖 | ① Raw 质控 |
| `config/join_rules.json` | 表间关联规则 | ② 逻辑关联 |
| `config/field_edits.json` | 逐列清洗意见 + 审阅/应用历史 | ③ AI 预处理 |
| `output/*_clean.csv` | 清洗结果 | ③ 应用时写 |
| `profile.json` | 字段画像缓存（`{updated_at, fields[]}`） | 扫描时写 |
| `meta.json` | 项目元信息（名/来源/文件列表/时间） | 创建与更新 |

- **无服务端会话**：当前项目完全由前端在请求头 `X-Project-Id` 传递。`app.require_project()` 仅校验该 id 对应目录存在，**不持有"当前项目"全局状态**——所以多标签页/多项目互不串扰。
- `load_json_config` / `save_json_config` 是项目级 JSON 读写的统一入口。

## 3. 字段画像引擎（`qc_engine.py`）

画像是整个工具的地基，③④⑤ 都依赖它。

### 3.1 类型推断 `infer_field_stats`
一律以**字符串**读入（`dtype=str`），再判定（阈值见代码）：
```
forced_type 优先 → 空列 → 日期(可转率≥0.9且非空≥5) → 数值(可转率≥0.9且非空≥5)
→ 分类(去重≤50 或 唯一率<0.01) → 否则 文本
（ID 型不自动判定，由用户手动覆盖）
```
返回 `non_null / null / null_pct / unique / inferred_dtype / variable_content / value_domain`，数值另带 `numeric_stats`，日期带 `date_stats`。

### 3.2 值域 `build_content_and_domain`
按类型给不同 `value_domain`：分类/ID/短文本 → Top 频次列表（带占比）；数值/日期 → 摘要字符串。**改类型后调用的是同一函数**，因此界面"改类型即时重算值域"与初次扫描结果完全一致。

### 3.3 超大表采样（性能关键）
`DEFAULT_SAMPLE_ROWS` 由环境变量 `PREPROCESS_PROFILE_SAMPLE_ROWS` 控制（默认 20 万，`0`=全量）。`profile_all_tables` 对每张表 `read_csv(p, nrows=sample_rows)`；若读到的行数 ≥ 上限则标 `sampled=True` 并记 `sample_rows`。
> 背景：实测某 478MB / 400 万行表全量画像约 **344s**；采样后降到秒级。代价是缺失率/去重/频次为样本估计。

### 3.4 多编码兜底
`read_csv` 依次尝试 `utf-8-sig / utf-8 / gbk / gb18030`，适配国内医疗系统常见 GBK 导出。

## 4. 表间关联（`join_analyzer.py` + `join_ai.py`）

### 4.1 统计探查（无 AI）
`probe_join_candidates`：对候选 ID 类列各取样本值集合（`_sample_series`，上限 1.2 万行），两两计算**列值交集匹配率**（`_match_stats`），产出候选列对。纯统计、可解释，但同名不同义列可能误配。

### 4.2 AI 识别
`discover_join_rules_with_ai`：把 `build_schema_summary(fields, with_domain=True)`（含关键 ID 列**完整 top 值域**）+ 探查结果 + 各表前 2 行样例，喂给 LLM，要求输出固定 JSON（路径/规则/key_ids/pitfalls/mermaid）。
> `with_domain=True` 是专为关联场景加的：让 AI 看到真实取值（如 `患者ID: HSCT_0709(12988)…`）而非仅 80 字摘要，显著提升列对判断质量。无 mermaid 时用 `build_simple_mermaid` 兜底生成。

## 5. 清洗：从意见到结果（`field_preprocess.py` + `transform_engine.py`）

这是"AI 不执行代码"原则的落点。流程分**草稿 → 审阅 → 应用**三态。

### 5.1 状态机（`config/field_edits.json`）
每张表一个 cfg：`{raw_file, output_file, table_note, fields:{列名:{instruction, source, status}}, history, review_status}`。
- `source`：`ai`（AI 草稿）/ `user`（人改）
- `status`：`draft` → `applied`
- `review_status`：`ai_draft` → `reviewed`

### 5.2 生成草稿
`run_ai_fill_table` → `ai_transformer.ai_suggest_field_edits`：传画像+表注+关联规则+当前列名，AI 为每列回一句意见；缺的列补「保留」。

### 5.3 编译为操作
应用时 `prepare_and_apply_table`：
- **有 AI**：`ai_compile_field_operations` 把自然语言意见 + 3 行样例编译成 JSON 操作列表。
- **无 AI**：`compile_operations_from_edits` 用正则 `_parse_instruction` 识别「改名为/删除/合并为/去空格」等中文意图；无法解析则报错提示。

### 5.4 安全执行（白名单）
`transform_engine.apply_operation` 是一个 **action 分发器**，只认 9 种操作：
`rename_column · drop_columns · merge_columns · fillna · map_values · strip_whitespace · dedupe · filter_rows · select_columns`，未知 action 直接 `raise`。**全程 pandas DataFrame 变换，无 `eval`/`exec`**，因此 LLM 即便越权也无法执行任意代码。

### 5.5 raw 不可变 + 可迭代
`resolve_working_source`：若 `output/` 已有该表清洗结果，则**以 output 为输入**继续；否则读 raw。结果永远写 output。删掉 output 文件即回到初始态。`apply_table` 写 `utf-8-sig`（Excel 友好）并记 `history`。

## 6. 数据分析（`analysis.py`）

`column_distribution(path, field, sample_rows)`：采样读单列，按 90% 数值率+去重>20 判为数值 → `numpy.histogram` 出直方图 bins；否则取 Top-20 类别频次。供 ⑤ 的 Chart.js 渲染。
> 对比视图（raw vs 处理后）在**前端**完成：分别请求两个 scope 的 `/api/analysis/fields` 与 `/distribution`，在 JS 里 diff 列集合与缺失率、并排画分布——后端因此只需两个无状态端点。

---

## 7. AI 配置同源解析（`ai_settings.py`）

`resolve_ai_config` 把 `api_key / base_url / model` 作为**一个整体**按来源取，杜绝跨源拼装：

```
1. 若 PREPROCESS_AI_API_KEY 存在 → 整组取该环境变量（base/model 同前缀）
2. 否则若本地文件有 key       → 整组取 ai_settings.local.json
3. 否则若 OPENAI_API_KEY 存在 → 整组取 OPENAI_* 通用变量
4. 否则 未配置
```
> 修复的历史 bug：旧逻辑 key 与 base_url 各自独立挑来源，导致"OpenAI 的 key + 本地文件里的 DeepSeek url" → 401。同源解析后，界面选了哪个平台就整组用哪个。

## 8. API 速览（`server/app.py`）

| 方法 路径 | 作用 |
|---|---|
| `GET /api/health` | 健康检查 + AI 是否就绪 |
| `GET/PUT /api/ai/settings`，`POST .../test` | 读写/测试 AI 配置 |
| `GET /api/projects`，`POST .../from-path`，`.../upload` | 项目列表/创建 |
| `GET /api/raw/tables \| profile \| field-types` | 表清单 / 画像 / 类型枚举 |
| `POST /api/raw/scan` | 全量扫描生成画像 |
| `POST /api/raw/type-override[/batch]` | 改类型（重算/批量保存） |
| `GET /api/raw/profile/export` | 下载质控画像 CSV |
| `GET/PUT /api/join-rules`，`POST .../probe \| discover` | 关联规则 / 探查 / AI 识别 |
| `GET /api/preprocess/tables \| fields`，`PUT .../edits` | 清洗列表/字段/存意见 |
| `POST /api/preprocess/ai-fill-table\|all`，`apply-table\|all` | 生成草稿 / 应用 |
| `GET /api/analysis/fields \| distribution` | 数据分析（type/missing、列分布） |
| `GET /api/output/files`，`.../download/{name}` | 导出列表/下载 |

> 除少数带 `project_id` 查询参数的下载链接外，需项目的端点统一通过请求头 `X-Project-Id` 标识项目。

## 9. 扩展指引

- **加一种清洗操作**：在 `transform_engine.apply_operation` 增一个 `action` 分支 → 在 `ai_transformer` 的 SYSTEM_PROMPT 操作清单里加一行 → （可选）`_parse_instruction` 加无 AI 兜底。
- **加一种类型/值域**：改 `qc_engine.infer_field_stats` 判定与 `build_content_and_domain` 展示，前端 `TYPE_META` 加显示名。
- **加一个分析图**：后端 `analysis.py` 加分布函数 + `app.py` 端点；前端 `panel-5` 加图表（复用 Chart.js）。
- **换 AI 平台**：任何 OpenAI 兼容 `/v1/chat/completions` 接口，界面选「其他」填 base_url 即可，代码无需改。
